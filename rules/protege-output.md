# プロテジェ効果（学習アウトプット）

## 基本方針

タスク完了時にユーザーが「誰かに教える形」で理解を言語化する機会を作る。
「説明できる = 理解している」の自己確認を促し、受動的知識を能動的知識に変換する。

## 発動タイミング

タスクが完結したタイミング（作業中ではない）に ★ Protégé ブロックを1つ出す。

## 発動条件

以下のいずれかに該当し、かつ学習モードが ON のとき:
- 設計判断・トレードオフを含む変更があった
- 原因分析（デバッグ・調査）で根本原因を特定した
- ★ Insight が生成されたタスクだった

## スキップ条件

- 学習モードが OFF（「全部やって」「急ぎ」「任せる」等）
- ボイラープレートのみの変更（import、型定義、設定ファイル等）
- 軽微な変更（typo修正、コメント修正）
- セッション内で既に多用している（連続ターンでは出さない）

## learning-mode.md との共存

途中の「考える」問いかけと完了後の「言語化する」は性質が異なるため、同タスク内で両方発動してよい。
ただし1タスク内の合計停止回数（learning-mode + Protégé）が3回を超えないこと。

## 質問プロンプト（review-and-design.md）との棲み分け

★ Protégé が発動する場合、質問プロンプトは省略する（同ターンでの認知負荷重複を防ぐ）。

## 問いかけフォーマット

タスク種別に応じた問いかけを1つ投げる。1〜3行の言語化を求める。

### 設計判断の場合

```
★ Protégé ─────────────────────────────────────
今回 [X ではなく Y] を選んだ。

「なぜ Y にしたの？」と聞かれたら、トレードオフを含めてどう説明する？
─────────────────────────────────────────────────
```

### 原因分析の場合

```
★ Protégé ─────────────────────────────────────
[問題] が [根本原因] で起きていた。

この因果関係を、ログを見ていない後輩にどう説明する？
─────────────────────────────────────────────────
```

### 実装判断の場合

```
★ Protégé ─────────────────────────────────────
今回の実装で核心の判断は [該当箇所] だった。

なぜその方法にしたか、あとで読み返せるように一言で書いてみて。
─────────────────────────────────────────────────
```

## 問いかけ後のふるまい

- 問いかけた後は **停止する** — ユーザーが回答するまで先に進まない
- 良い説明: 「その通り。特に [X] が本質を捉えている」とコンパクトに返す
- 不完全な説明: 「その点は正しい。[Y] はどう説明する？」と掘り下げる
- 的外れな説明: 正解を言わず「[Z] に着目すると？」と誘導する

## 記録

ユーザーが回答したら `tasks/learning-journal.md` に追記する（未存在なら新規作成）:

```
## [YYYY-MM-DD] | [タスク概要]

**問いかけ:** [Protégéの問いかけ内容]

**自分の説明:**
[ユーザーの回答をそのまま記録]

**補足:**
[AIのフィードバック、なければ省略]
```
