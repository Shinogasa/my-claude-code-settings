## 2026-04-17 | プロテジェ効果ルールの配置設計

**問いかけ:** プロテジェ効果のルールを既存ルールへの追記ではなく、独立した新規ファイルとして配置した理由は？

**自分の説明:**
今回の追加は作業中での問いかけによる「考える」という行為とは別軸のもので、作業完了後に学びを復習するものである。だから別物として独立したルール設定とした。

**補足:**
単一責任原則がルールファイル設計にも適用される好例。タイミング（途中 vs 完了後）と認知プロセス（思考 vs 言語化）の両方が異なるため、独立ファイルが正しい。

## 2026-05-15 | settings.local.json をコミットに含めるかの判断

**問いかけ:** `.claude/settings.local.json` をコミットに含めず `.gitignore` へ追加という判断をした。「なぜ含めなかったの？」と聞かれたら、トレードオフを含めてどう説明する？

**自分の説明:**
個人設定かつ端末依存のパスが入っているため含めないのが良い。

**補足:**
特に「端末依存のパス」という具体性が判断の決定打。別端末で clone した際に自分のパスを上書きしてしまい、`Read` permission 等が機能しなくなる障害に直結する。Claude Code 慣例として `settings.json`(共有) / `settings.local.json`(個人) の二段構成自体が「個人差を吸収する設計パターン」であり、その慣例とも整合した判断。
