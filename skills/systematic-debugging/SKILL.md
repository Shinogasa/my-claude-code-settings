# Systematic Debugging

## 概要

当てずっぽうの修正は時間を浪費し、新たなバグを生む。

**鉄則: 根本原因の特定なしに修正を試みてはならない。**

Phase 1を完了していなければ、修正を提案できない。

## いつ使うか

あらゆる技術的問題に使用する:
- テスト失敗
- 本番バグ
- 予期しない動作
- パフォーマンス問題
- ビルド失敗
- 外部連携の問題

**特に以下の状況で必ず使う:**
- 時間的プレッシャーがあるとき（急ぐほど当てずっぽうに走りやすい）
- 「明らかな修正」が見えたとき（それは症状への対処かもしれない）
- すでに複数の修正を試して失敗しているとき

## 4フェーズプロセス

各フェーズを順番に完了すること。スキップ不可。

### Phase 1: 根本原因の調査

**修正を試みる前に、以下をすべて実行する:**

1. **エラーメッセージを精読する**
   - スタックトレースを最後まで読む
   - 行番号、ファイルパス、エラーコードを記録する
   - 警告も見逃さない

2. **一貫して再現する**
   - 確実にトリガーできるか？
   - 正確な再現手順は？
   - 再現できない場合 → データ収集を続ける。推測しない

3. **直近の変更を確認する**
   - `git diff`、直近のコミット
   - 新しい依存関係、設定変更
   - 環境の差異

4. **マルチコンポーネントシステムでは証拠を収集する**
   - 各コンポーネント境界でデータの入出力をログに出す
   - 環境変数・設定の伝播を確認する
   - どのレイヤーで壊れているかを特定してから、そのコンポーネントを調査する

5. **データフローをトレースする**
   - 不正な値はどこで発生したか？
   - 何がその値を渡したか？
   - 源流まで遡る。症状の場所で直さない

### Phase 2: パターン分析

1. **動作する類似コードを探す**
   - 同じコードベース内の類似実装
   - 動いているものと壊れているものの差異をリストアップ

2. **リファレンスと比較する**
   - パターンを実装しているなら、リファレンスを**完全に**読む
   - 斜め読みしない。全行読む

3. **差異を特定する**
   - 「これは関係ない」と決めつけない
   - 小さな差異もすべてリストアップ

### Phase 3: 仮説と検証

1. **仮説を1つ立てる**
   - 「Xが根本原因だと考える。なぜならY」と明文化する
   - 曖昧にしない

2. **最小限の変更でテストする**
   - 1つの変数だけ変える
   - 複数の修正を同時にしない

3. **結果を確認してから進む**
   - 成功 → Phase 4へ
   - 失敗 → **新しい**仮説を立てる。同じ修正を重ねない

### Phase 4: 修正の実装

1. **失敗するテストケースを作成する**
   - バグを再現する最小限のテスト
   - テストなしの修正は禁止
   - tdd-workflowスキルと連携する

2. **単一の修正を実装する**
   - 根本原因に対処する変更のみ
   - 「ついでに」の改善を混ぜない

3. **修正を検証する**
   - テストがパスするか？
   - 他のテストが壊れていないか？
   - verification-loopスキルで全フェーズ確認

4. **3回修正に失敗したら → アーキテクチャを疑う**
   - 各修正が別の場所で新たな問題を露出する → 設計の問題
   - 4回目の修正に進まない
   - 設計の見直しについてユーザーと議論する

## 合理化防止（Rationalization Prevention）

以下の思考が浮かんだら、それはプロセスをスキップする合理化である。立ち止まること。

| 合理化 | 現実 |
|---|---|
| 「シンプルなバグだからプロセス不要」 | シンプルなバグにも根本原因がある。プロセスは軽いバグなら数分で終わる |
| 「緊急だから調査を省略」 | 当てずっぽうの修正を繰り返す方が遅い。体系的調査の方が速い |
| 「まずこれを試して、ダメなら調査する」 | 最初の修正がパターンを決める。最初から正しくやる |
| 「原因は見えている、すぐ直せる」 | 症状が見えている≠根本原因を理解している |
| 「複数まとめて直した方が効率的」 | 何が効いたか分離できない。新たなバグを生む |
| 「リファレンス長すぎるから要点だけ」 | 部分的な理解はバグを保証する。全部読む |
| 「テストは後で書く」 | テストなしの修正は定着しない。先にテストを書く |
| 「もう1回だけ修正を試す」（2回以上失敗後） | 3回以上の失敗＝アーキテクチャの問題。修正ではなく設計を見直す |

## クイックリファレンス

| フェーズ | 主な活動 | 完了条件 |
|---|---|---|
| **1. 根本原因** | エラー精読、再現、変更確認、証拠収集 | 何が・なぜ起きているか理解 |
| **2. パターン** | 動作例の発見、差異の比較 | 壊れている箇所の特定 |
| **3. 仮説** | 仮説の明文化、最小限テスト | 仮説の確認 or 新仮説 |
| **4. 修正** | テスト作成、修正、検証 | バグ解消、テスト全パス |
